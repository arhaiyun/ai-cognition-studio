# Stage 3: PRD 文档生成

**只有 Stage 2 + Stage 2.5 都确认后才能写。**

PRD 必须能直接交给设计师 / 前后端 / AI 编程工具使用。

## PRD 结构

```markdown
# <项目名> PRD

## 文档信息
- 版本 / 作者 / 创建日期 / 最后更新

## 1. 项目背景
- 业务背景 / 用户问题 / 解决方案立场

## 2. 产品目标
- 业务目标（可量化）
- 用户价值
- 成功指标（公式 + 测量方式 + 目标值 + 频率）

## 3. 用户角色
- 角色 1：画像 + 核心场景 + 关键诉求
- 角色 2：...

## 4. 需求范围
- ✅ 本次 MVP 范围
- ❌ 明确不做（每条带理由）
- 📌 未来版本（v2 / v3）

## 5. 功能清单
- 功能 ID / 名称 / 优先级 / 简述 / 验收要点

## 6. 页面说明
每个页面：
- 页面 ID / 名称 / URL / 入口 / 出口
- 页面目的（用户来这里要完成什么）
- 主要模块
- 数据来源

## 7. 用户流程
关键流程画成 HTML+SVG 流程图（不用 Mermaid，不用 ASCII），存到 `prd-assets/flow-<name>.html`

## 8. 功能详细说明
每个核心功能：
- 触发条件
- 前置条件
- 主流程步骤
- 分支与异常
- 操作结果
- 涉及数据
- 权限要求

## 9. 字段与状态
- 核心 entity 的 schema：字段名 / 类型 / 约束 / 默认值 / 示例 / 说明
- 状态机：状态列表 + 状态转移条件（画成 HTML+SVG）

## 10. 权限规则
- 权限矩阵：角色 × 资源 × 动作

## 11. 异常与边界
- 网络异常 / 数据为空 / 权限不足 / 数据冲突 / 输入非法 …… 每个场景的 UI 反馈与系统行为

## 12. 埋点建议
- 核心事件清单 + 触发时机 + 上报字段

## 13. 非功能需求
- 性能：首屏加载 / 接口响应 / 并发
- 兼容性：浏览器 / 设备 / 屏幕
- 安全：认证 / 数据加密 / 防刷
- 可观测：日志 / 监控 / 告警
- 国际化 / 无障碍

## 14. 验收标准
- 每个核心功能的 Given-When-Then 验收用例

## 15. 版本规划
- MVP / v2 / v3 时间与功能

## 16. 未决问题
- 真·待确认事项（不是甩锅）

---

## 专家自检段
（强制）资深产品经理会怎么挑刺这份 PRD？至少 3 条。
```

## 重要约束

- PRD 出现的每个核心概念，必须在文档内有数学定义/schema/边界值，不能停在术语层
- PRD 完成后，所有 HTML 流程图 / 状态机图必须实际产出到 `prd-assets/` 目录
- PRD 长度不限，但每一段都要"有信息"，不要凑字数
- 写入 `prd/<name>-prd.md`
