# Stage 6: 3 个 Cursor / Codex 实现 Prompt

**最终产出物。**

每个 Prompt 必须是**自包含、可复制、可直接执行**的实现说明。
Cursor / Codex 拿到 Prompt 不应该需要回来问任何问题。

## 产出 3 份独立 Markdown 文件

```
prompts/
├── prompt-1-init.md        # 项目初始化与基础架构
├── prompt-2-core.md        # 核心功能实现
└── prompt-3-polish.md      # 完善体验、测试与交付
```

## 每份 Prompt 必须包含的 9 段

```markdown
# Prompt N: <名称>

## 1. 实现目标
（一段话说清楚这个阶段要达成什么，验收时的可见状态是什么）

## 2. 背景上下文
（给 AI 解释这是什么项目、为什么做、目标用户是谁。摘取 PRD 的最关键部分）

## 3. 输入资料引用
- PRD 文档：`prd/<filename>.md`
- UED Demo：`demo/index.html` 及子页面
- 技术实现方案：`tech/<filename>.md`
- 数据 schema：`tech-assets/erd.html`
- 流程图：`prd-assets/flow-*.html`

（必须列实际文件路径，不要写"参考之前的设计文档"这种话）

## 4. 具体任务列表
按顺序的 task checklist。每条：
- T<N>: <动作> — <具体命令 / 文件路径>
- 输入 / 输出 / 验证方式都要明确

## 5. 文件与模块要求
- 哪些文件必须创建
- 文件命名规范
- 模块组织约定

## 6. 技术约束
- 必须用的库与版本
- 必须遵守的代码风格
- 必须避免的反模式

## 7. 验收标准
具体到可执行的检查：
- 命令 X 跑通
- 页面 Y 能在浏览器打开且看到 Z
- 测试用例 W 全通过

## 8. 不允许做的事
- ❌ 不要重新设计 UI，必须严格复刻 demo/<page>.html
- ❌ 不要改 PRD 里的字段名 / 状态机
- ❌ 不要加 PRD 里没要求的功能
- ❌ 遇到歧义不要自己决定，先在汇报里列出问题
- （其他领域专属禁止项）

## 9. 完成后的汇报格式
要求 AI 完成后按下面格式汇报：
- ✅ 已完成 task：T1, T2, ...
- ⚠️ 遇到的歧义与决策：...
- 🐛 已知问题：...
- 📁 关键文件清单：...
- ▶️ 如何运行验证：...
```

## 三份 Prompt 的具体边界

**Prompt 1: 项目初始化与基础架构**
- 范围：技术栈搭建 / 目录结构 / 路由 / 全局样式系统 / 基础组件库 / 开发规范 / 代码风格配置 / Git hooks / 一个能跑起来的空壳
- 验收：`pnpm dev`（或对应命令）能起，能看到一个空页面，无报错

**Prompt 2: 核心功能实现**
- 范围：数据模型 / 数据库 migration / API 实现 / 核心页面 / 业务流程闭环 / 状态管理 / 权限逻辑 / 主要交互
- 验收：PRD §5 功能清单里所有 P0 功能可走通完整 happy path

**Prompt 3: 完善体验、测试与交付**
- 范围：UI 细节打磨 / 异常状态 / 空状态 / 错误处理 / 响应式 / 性能优化 / 单元测试 + E2E / 部署脚本 / README / 用户文档
- 验收：测试通过率 ≥ 目标值 / 性能指标达标 / 能部署到 staging

## 完成 3 份 Prompt 后告知用户

「3 份 Prompt 已生成在 `prompts/` 目录。建议执行顺序：1 → 2 → 3。可以把这些 Prompt 直接复制给 Cursor / Codex。」
