# Stage 2: 产品方案设计（标注"待确认"）

基于访谈信息提出产品方案，所有结论都标注"待确认 ⚠️"。

## 必须包含（顺序自定，按领域调整侧重）

1. **领域语境与一句话立场**：这个产品的核心信念是什么，一句话
2. **产品定位**：是 X 而不是 Y，因为 Z
3. **用户画像**：具体到一个人，配场景剧本
4. **核心用户旅程**：从触发点到完成动作的完整路径（用文字段落描述，不要扁平 bullet）
5. **功能模块拆分**：模块树
6. **MVP 功能范围**：明确 ✅做 与 ❌不做 + 每条理由
7. **后续版本规划**：v2 / v3 暂存
8. **页面地图**：所有页面的层级关系
9. **关键业务流程**：核心 happy path + 关键分支
10. **核心数据对象**：entity 列表 + 关键字段
11. **权限角色**：角色清单 + 各自能力边界
12. **风险点与待确认问题**：真正需要用户拍板的取舍

## 质量红线（违反任何一条都重写）

- ❌ 不准用"流畅 / 好用 / 提升体验"这类空词，必须用数字/阈值/公式/边界值
- ❌ 不准把推理塞进 bullet list（推理用段落，bullet 只用于"枚举对等项"）
- ❌ 不准用表格走推理（表格只用于：参数空间 / schema / 多选项对比）
- ❌ 不准把可以自己拍板的题塞进"待确认"丢回给用户
- ❌ 不准用"建议 / 可以考虑 / 也许"语气，写"用 X，不用 Y，因为 Z"

## 写完必须做的事

**专家自检段**：假设我把这份方案给一个 10 年该领域经验的资深从业者看，他会指出哪些"想当然 / 浅了 / 没考虑"的地方？至少 3 条具体批评 + 改进动作。

## STOP 主动问

「以上范围是否确认？哪些功能要删减、增加或调整？哪些待确认问题需要现在拍板？」

**用户必须显式确认才进 Stage 2.5。**
