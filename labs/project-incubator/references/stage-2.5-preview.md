# Stage 2.5: 早期视觉预览 Demo（轻量原型，PRD 之前）

**目的**：用真实视觉/交互暴露分歧，**早于** PRD 文字细节工作 —— 避免 PRD 写完后才发现核心页面布局或通知格式不对，反复返工。

**这不是正式 Stage 4 Demo**。规模刻意压小，只做"看一眼就能判断方向对不对"的程度。

## 必须产出

```
preview/
├── index.html              # 导航入口
├── <key-page-1>.html       # 主战场页面（最关键的 1 个）
├── <key-page-2>.html       # 次重要 1-2 个
└── <interaction-key>.html  # 关键交互形态（如通知卡片、单条信号、推送弹窗等）
```

页面数量 **3-5 个**，不超过 5 个。优先选 **主战场页面 + 决策最难的形态**（例如"通知优先"的项目，第一个一定是通知卡片样式）。

## 强制要求

- ✅ 真实文案 + 真实数据（用真实业务样例，例如真票名 NVDA / TSLA）
- ✅ 锁定风格锚点（在 Stage 1 已问，此处直接落地视觉调性）
- ✅ 关键交互能演示：列表点击 / Tab 切换 / 状态变化 至少各 1 个
- ✅ 单文件可打开（Tailwind CDN 可用，禁止 React/Vue）

## 刻意不做（留给正式 Stage 4）

- ❌ 不做完整设计 token（色板/字体/间距全集）—— 用近似值即可
- ❌ 不做空状态 / 错误状态 / 加载状态全集
- ❌ 不做响应式全断点 —— 只做最相关的 1-2 个
- ❌ 不做次要页面（配置页 / 管理页 / 列表 CRUD 等）

## 写完必须做的事

**专家自检段**：3 条 + 在 README 里明确列出"本预览没做但正式 Stage 4 会做的事"（避免用户误以为 Stage 4 就是这个程度）。

## STOP 主动问

「这个视觉方向和关键交互形态确认吗？哪个页面或交互需要重画？」

**用户必须显式确认才进 Stage 3。视觉/交互方向锁定后，Stage 3 PRD 才有意义。**
